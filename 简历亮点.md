# MedFlow-AI 简历亮点

## 1. 使用说明

本文件用于集中维护该项目在简历、面试、自我介绍和项目介绍中的技术亮点表述。

适用场景：

- 简历项目描述
- 面试项目亮点回答
- 自我介绍中的项目概述
- 项目复盘和技术沉淀

维护原则：

- 优先写技术亮点，不写空泛业务描述
- 优先写“为什么这么做”和“解决了什么问题”
- 优先写可追问、可展开、可落地的内容
- 避免堆砌名词，强调工程化能力

---

## 2. 项目一句话描述

MedFlow-AI 是一个基于 Spring Boot、LangChain4j、Vue 3 和 Pinecone 构建的医疗 AI 助手项目，具备多轮会话记忆、RAG 知识检索、流式响应和工具调用能力，并围绕模型接入、检索增强、可观测性和稳定性做工程化设计。

---

## 3. 简历高频亮点

以下内容适合直接写入简历。

### 亮点 1

基于 LangChain4j 构建医疗问诊 Agent，整合系统提示词、会话记忆、RAG 检索和工具调用能力，实现医疗咨询场景下的多轮智能对话。

### 亮点 2

设计多模型接入方案，兼容 OpenAI Compatible、DashScope 和 Ollama，本地模型与云端模型可灵活切换，提升系统扩展性和环境适配能力。

### 亮点 3

基于 Pinecone + Embedding 构建医疗知识检索链路，将医院信息、科室资料和医学文档向量化，提升问答结果的相关性与专业性。

### 亮点 4

通过 MongoDB 持久化对话记忆，结合 `memoryId` 实现多轮上下文关联，提升连续问诊场景下的交互体验和上下文一致性。

### 亮点 5

封装预约挂号、取消预约、号源查询等工具能力，打通 AI 对话与业务动作执行流程，体现 Agent 与传统业务系统联动能力。

### 亮点 6

前端基于 Vue 3 + Vite 实现流式消息渲染，支持增量输出、自动滚动和会话管理，优化大模型对话类产品的用户体验。

## 5. 面试高含金量技术亮点

以下内容建议作为后续项目迭代重点，也是面试里最容易体现技术深度的方向。

### 5.1 模型抽象层

目标：

- 将不同模型供应商的调用方式统一封装
- 降低业务层对具体模型平台的耦合
- 支持模型切换、灰度和降级

面试表达：

- 不是简单调用某个模型 SDK，而是设计了统一模型适配层
- 对业务侧暴露统一的聊天、流式输出和 embedding 接口
- 使得 DashScope、Ollama、OpenAI Compatible 模型可以无感切换

可包装关键词：

- 模型网关
- 模型适配层
- 统一模型接口
- 多模型治理

### 5.2 RAG 检索优化

目标：

- 从“能检索”升级到“检索更准”
- 提升回答相关性和知识命中率
- 降低无关上下文对回答的污染

建议迭代点：

- 查询改写
- 文档切片策略优化
- TopK 动态调整
- 检索重排
- 引用来源返回

面试表达：

- 围绕医疗知识问答场景优化 RAG 链路
- 通过查询重写和召回策略优化提升知识命中率
- 通过来源片段返回增强回答可信度

### 5.3 可观测性建设

目标：

- 看得见模型调用链路
- 定位性能瓶颈和失败原因
- 支撑后续优化与稳定性治理

建议迭代点：

- traceId
- 模型耗时
- 检索耗时
- tool 调用耗时
- 错误分类
- token 消耗统计

面试表达：

- 为 AI 请求链路设计可观测体系
- 能区分模型慢、检索慢还是工具执行慢
- 为性能优化和线上排障提供数据支撑

### 5.4 稳定性与降级机制

目标：

- 提高 AI 系统在异常情况下的可用性
- 避免因模型、检索或工具失败导致整体不可用

建议迭代点：

- 主模型失败自动切换备用模型
- 检索失败降级为纯模型回答
- 工具调用失败兜底
- 超时控制和重试
- 流式中断保护

面试表达：

- 针对 LLM 系统不稳定问题设计工程化容错机制
- 将 AI 系统按生产级服务的标准进行可用性治理

### 5.5 Prompt 工程治理

目标：

- 避免提示词分散、硬编码、不可管理
- 提升提示词迭代效率

建议迭代点：

- Prompt 模板化
- Prompt 变量化
- 多场景 Prompt 切换
- Prompt 版本管理

面试表达：

- 不只是写提示词，而是做了 Prompt 工程治理
- 提高不同场景下 AI 行为控制的稳定性和维护效率

## 8. 面试表达模板

### 8.1 项目介绍模板

这个项目本质上是一个医疗 AI 助手，我主要做的是把大模型能力和传统业务能力整合起来。技术上它不是单纯的聊天应用，而是把多轮会话记忆、RAG 检索、流式输出和工具调用编排到一起，让系统既能回答知识类问题，也能执行预约这类业务动作。

### 8.2 技术亮点模板

我觉得这个项目最有技术含量的部分，不是接了某个模型，而是把模型调用、知识检索、上下文记忆和业务工具统一组织成一个可扩展的 Agent 链路。后续如果继续打磨，我会重点增强模型抽象层、RAG 优化、可观测性和失败降级，因为这些更能体现 AI 系统工程化能力。

### 8.3 难点模板

项目里比较典型的难点是，大模型应用并不是“接口调通就行”，而是需要处理上下文一致性、知识召回质量、业务动作执行和异常兜底这些问题。我在这个项目里更关注的是把这些能力做成结构化、可维护、可扩展的系统。
